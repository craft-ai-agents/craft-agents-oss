Refactor the auth middleware to remove the global token registry leak — and add a regression test that fails on the leaked path.
§ 03 agent · craft 14:32:09 EDT — streaming

The leak lives in packages/server/src/auth/middleware.ts at line 47, where tokenRegistry is declared at module scope. Every request reads and writes the same Map, so a token issued for tenant A briefly becomes visible to tenant B during the gap between issuance and revocation. Calling it a leak is generous — it's a cross-tenant data path that the type system happily endorses.

The right fix is to scope the registry to the request, not the module, and to add a regression test that proves cross-tenant tokens cannot leak even when the same process handles both. I'll keep the public surface of requireAuth identical so the route handlers don't have to change. The Map moves inside a factory keyed by tenantId, and the cleanup on response.end stays where it is.

The test is the load-bearing piece. Without it, this fix can silently regress — the symptom only appears under concurrent traffic from two tenants, which doesn't happen in unit tests. I'll spin up two requests in parallel against the patched middleware and assert the registries are separate object identities, then assert neither token is visible to the other.

One thing worth flagging: the existing cleanupOnResponseEnd hook (line 112) assumes a single global Map and calls tokenRegistry.delete(token). After the change, that delete is a no-op because the per-request Map is already garbage-collected when the response closes. The hook can be removed, but I want to keep it for one release as a safety net — in case any caller is mutating registry state directly through the export. We can drop it in the next minor version once telemetry confirms zero hits.

Three edits in total: the middleware itself, a new isolation test, and an updated changelog entry that names the CVE-class for downstream auditors. ETA on apply is about eighteen seconds
